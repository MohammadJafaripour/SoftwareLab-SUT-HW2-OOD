# فاز دوم

| اصل                         | وضعیت          | علت برقراری / نقض                                                                                                                                                                                                |
| :-------------------------- | :------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **SRP** (تک‌مسئولیتی)       | **نقض شده**    | کلاس `ReservationService` وظایف متعددی دارد: محاسبات تخفیف شهرها، منطق انتخاب درگاه پرداخت، فرمت‌بندی متن فاکتور و مدیریت بستر ارسال اعلان‌ها. تغییر در هر کدام از این نیازمندی‌ها باعث دستکاری این کلاس می‌شود. |
| **OCP** (باز/بسته)          | **نقض شده**    | کد نسبت به توسعه بسته است. برای افزودن یک روش پرداخت یا یک بستر پیام‌رسانی جدید، ناچاریم ساختارهای کنترلی `switch-case` را در کلاس `ReservationService` تغییر دهیم.                                              |
| **LSP** (جانشینی لیسکوف)    | **برقرار است** | کلاس `LuxuryRoom` بدون تغییر دادن رفتار اصلی یا نقض پیش‌فرض‌های کلاس پایه (`Room`)، ویژگی‌های خود را اضافه کرده است و می‌توان آن را بدون مشکل جانشین کلاس والد کرد.                                              |
| **ISP** (تفکیک اینترفیس)    | **نقض شده**    | اینترفیس `MessageSender` حاوی متد اختصاصی `sendEmail` است. وقتی کلاس `SmsSender` بخواهد آن را پیاده‌سازی کند، مجبور به پذیرش متد مربوط به ایمیل می‌شود که ارتباطی با پیامک ندارد.                                |
| **DIP** (وارونگی وابستگی)   | **نقض شده**    | کلاس high-level ما یعنی `ReservationService` به جای وابستگی به انتزاع (Interfaces)، مستقیماً به کلاس‌های انضمامی low-level مانند `PaymentProcessor` و `EmailSender` وابسته شده است (`new` شده‌اند).              |
| **PLK** (قانون دیمیتر)      | **نقض شده**    | خط `res.room.price *= 0.9;` زنجیره‌ای از دسترسی‌ها را نشان می‌دهد. کلاس سرویس نباید از ساختار داخلی رزرو به اتاق برسد و مستقیماً متغیر عمومی قیمت را دستکاری کند. این کار کپسوله‌سازی را نقض می‌کند.             |
| **CRP** (بازاستفاده ترکیبی) | **نقض شده**    | اگرچه از ترکیب (Composition) استفاده شده، اما به دلیل هاردکد شدن کلاس‌های انضمامی درون سرویس اصلی، امکان تزریق پویای وابستگی‌ها وجود ندارد و قابلیت بازاستفاده منعطف به شدت محدود است.                           |
